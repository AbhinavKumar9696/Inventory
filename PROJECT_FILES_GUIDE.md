# Project Files Guide

This file explains the purpose of the files and folders used in this project, how they connect to each other, and which ones are source code versus generated/runtime files.

## 1. What this project is

This is a backend inventory management API built with:

- FastAPI for HTTP endpoints
- SQLAlchemy async ORM for database access
- PostgreSQL as the main database
- Alembic for schema migrations
- Redis for caching and Celery broker/backend
- Celery for background jobs
- Poetry for dependency management
- Docker and Docker Compose for containerized development

At a high level, the request flow is:

1. A request enters a FastAPI route in `app/api/v1/`
2. The route uses dependencies from `app/core/dependencies.py`
3. A service in `app/services/` applies business rules
4. A repository in `app/repositories/` reads/writes database models from `app/models/`
5. Pydantic schemas in `app/schemas/` shape request and response data
6. Redis caching and Celery tasks are used where needed

## 2. Top-level files and folders

### `.env`

This is the local environment configuration file actually used when the app runs on your machine or in Docker Compose.

It typically contains:

- app settings like project name and debug mode
- database connection settings
- Redis connection settings
- JWT auth settings
- default admin seed credentials

Important note:

- this file may contain real secrets or local credentials
- it should be treated as private configuration, not public documentation

### `.env.example`

This is the safe template for `.env`.

It shows which environment variables the project expects, such as:

- `PROJECT_NAME`
- `ENVIRONMENT`
- `DATABASE_URL`
- `REDIS_URL`
- `JWT_SECRET_KEY`
- `DEFAULT_ADMIN_EMAIL`
- `DEFAULT_ADMIN_PASSWORD`

Use this file as the reference when creating a new environment.

### `.dockerignore`

This tells Docker which files and folders should not be copied into the Docker build context.

It excludes things like:

- `.git`
- `.venv`
- `__pycache__`
- Python cache files
- build artifacts

This keeps Docker builds smaller and faster.

### `README.md`

This is the human-facing project introduction.

It explains:

- the tech stack
- how to run the project with Docker
- how to run it locally
- key API endpoints
- default admin behavior
- basic troubleshooting

This file is good for onboarding, while this guide is more focused on internal file-by-file explanation.

### `pyproject.toml`

This is the main Python project configuration file for Poetry.

It defines:

- package metadata
- Python version requirement
- app dependencies
- dev dependencies
- Ruff lint settings
- pytest settings

Key dependencies listed here include `fastapi`, `sqlalchemy`, `alembic`, `redis`, `celery`, `python-jose`, and `passlib`.

The current dev dependencies also include:

- `pytest`
- `pytest-asyncio`
- `httpx`
- `ruff`
- `mypy`

### `poetry.lock`

This is the lockfile generated by Poetry.

It pins exact dependency versions so installs are reproducible across machines.

You usually do not edit this manually.

### `Dockerfile`

This defines how the application image is built.

What it does:

- starts from `python:3.12-slim`
- installs build tools and Poetry
- copies `pyproject.toml` and `README.md`
- installs dependencies without the dev group
- copies the rest of the source code
- exposes port `8000`
- runs Uvicorn by default

This is used by both the API container and the Celery worker container.

### `docker-compose.yml`

This defines the multi-container local development environment.

It starts four services:

- `api`: FastAPI app via Uvicorn
- `worker`: Celery worker
- `postgres`: PostgreSQL database
- `redis`: Redis cache and queue backend

It also:

- loads `.env`
- maps ports to your machine
- mounts the project folder into `/app`
- creates named volumes for database and Redis persistence

### `alembic.ini`

This is Alembic's top-level configuration file.

It sets:

- the migration script location
- default SQLAlchemy URL
- logging behavior for migrations

In practice, the database URL here is overridden by `app/core/config.py` inside `alembic/env.py`.

### `celery_worker.py`

This is a tiny entry-point file for Celery workers.

It imports `celery_app` from `app/tasks/celery_app.py` and exposes it so the command below works:

```powershell
celery -A celery_worker.celery_app worker --loglevel=INFO
```

### `.docker-tmp/`

This looks like a Docker-related temporary/runtime folder, not application source code.

The files inside it, such as `.token_seed` and `config.json`, are not part of the backend logic. They appear to be local tooling artifacts.

### `__pycache__/`

This top-level folder contains compiled Python bytecode (`.pyc`) generated automatically by Python.

It is not source code and can usually be ignored.

## 3. `app/` package overview

The `app/` folder contains the actual backend application.

### `app/__init__.py`

Marks `app` as a Python package. The docstring simply labels it as the inventory management backend package.

### `app/main.py`

This is the FastAPI application entry point.

It does several important things:

- configures logging early
- loads settings
- creates the FastAPI app
- registers custom exception handlers
- includes the versioned API router
- defines the application lifespan

The lifespan function is especially important because it:

- checks Redis connectivity on startup with `redis_client.ping()`
- opens a DB session
- seeds the default admin user if it does not exist
- closes the Redis client on shutdown

This file is the main place where application startup behavior is coordinated.

## 4. API layer: `app/api/`

This layer defines HTTP endpoints.

### `app/api/__init__.py`

Only marks the folder as the API package.

### `app/api/v1/__init__.py`

Marks `v1` as the versioned API module. The project uses `/api/v1` as its default prefix.

### `app/api/v1/router.py`

This is the central API router aggregator.

It imports and includes the routers from:

- `health.py`
- `auth.py`
- `users.py`
- `items.py`
- `inventory.py`

This keeps `app/main.py` simple and makes versioned route composition easy.

### `app/api/v1/health.py`

Defines the simplest endpoint in the project:

- `GET /health`

It returns:

```json
{"status": "ok"}
```

This is useful for smoke checks and container health validation.

### `app/api/v1/auth.py`

Defines authentication routes under `/auth`.

Main endpoints:

- `POST /auth/register`
- `POST /auth/login`

Important details:

- registration is protected with role checks, so only `admin` or `manager` users can create users
- login uses `OAuth2PasswordRequestForm`
- the OAuth2 form expects:
  - `username` to contain the email
  - `password` to contain the password
- the file depends on `AuthService` to perform the real logic

### `app/api/v1/users.py`

Defines user-related routes under `/users`.

Main endpoints:

- `GET /users/me`
- `GET /users/`
- `GET /users/{user_id}`
- `DELETE /users/{user_id}`

Behavior:

- `/me` returns the authenticated user
- `/users/` lists all users
- `/users/{user_id}` returns one user by id
- `DELETE /users/{user_id}` currently expects both:
  - the path parameter `user_id`
  - a `full_name` query parameter for extra confirmation
- role-based protection is applied using `require_roles(...)`

### `app/api/v1/items.py`

Defines inventory item CRUD-style endpoints under `/items`.

Main endpoints:

- `GET /items/`
- `POST /items/`
- `GET /items/{item_id}`
- `PATCH /items/{item_id}`

Behavior:

- listing and reading are open to authenticated staff, managers, and admins
- creation and update are limited to managers and admins
- this file delegates business logic to `ItemService`

### `app/api/v1/inventory.py`

Defines stock movement endpoints under `/inventory`.

Main endpoints:

- `POST /inventory/movements`
- `GET /inventory/items/total-quantity`
- `GET /inventory/items/{item_id}/movements`

Behavior:

- creates stock in/out records
- returns total quantity across all items
- reads movement history for a given item
- requires authentication and role checks
- gets the current user so `performed_by` can be stored correctly

## 5. Core utilities: `app/core/`

These files support config, auth, dependency injection, logging, and error handling.

### `app/core/__init__.py`

Marks the folder as the core helper package.

### `app/core/config.py`

This is one of the most important files in the project.

It defines the `Settings` class using `pydantic-settings`.

What it manages:

- project metadata
- environment name
- debug mode
- API prefix
- async database URL
- Redis URL and cache TTL
- JWT settings
- default admin seed credentials

Important detail:

- `sync_database_url` converts the async DB URL into a sync URL for Alembic by replacing `+asyncpg` with `+psycopg2`

Also important:

- `get_settings()` is wrapped with `@lru_cache`, so settings are loaded once and reused

### `app/core/dependencies.py`

This file wires together FastAPI dependency injection.

It provides factory functions for:

- `UserRepository`
- `ItemRepository`
- `MovementRepository`
- `AuthService`
- `UserService`
- `ItemService`
- `InventoryService`

It also defines authentication helpers:

- `oauth2_scheme`
- `get_current_user()`
- `require_roles(...)`

This is the bridge between routes and lower layers.

Runtime pattern:

- routes depend on services
- services depend on repositories
- repositories depend on DB sessions

### `app/core/exceptions.py`

Defines custom application error handling.

`AppException` stores:

- a human-readable error message
- an HTTP status code

`register_exception_handlers(app)` adds a FastAPI exception handler that converts `AppException` into JSON responses like:

```json
{"detail": "..."}
```

### `app/core/logging.py`

Sets up application logging using Python's `logging` module.

It:

- defines a single standard formatter
- sends logs to stdout/stderr through a stream handler
- sets the root level to `INFO`
- lowers SQLAlchemy engine logs to `WARNING`

This makes logs readable without too much SQL noise.

### `app/core/security.py`

Handles password hashing and JWT token work.

Functions here:

- `verify_password()`
- `hash_password()`
- `create_access_token()`
- `decode_access_token()`

Important behavior:

- passwords use `passlib` with `pbkdf2_sha256`
- access tokens include `exp` and `sub`
- token signing uses values from settings
- invalid JWTs raise a `ValueError`, which gets translated higher up into auth errors

## 6. Database layer: `app/db/`

This layer sets up SQLAlchemy base classes, sessions, and startup DB seeding.

### `app/db/__init__.py`

Marks the DB folder as a package.

### `app/db/base.py`

Defines the SQLAlchemy declarative base class:

- `Base`

All ORM models inherit from this.

This gives Alembic a shared metadata object for migrations.

### `app/db/session.py`

Creates and exposes shared database and Redis connections.

Main objects:

- `engine`
- `AsyncSessionLocal`
- `redis_client`

Dependency providers:

- `get_db_session()`
- `get_redis()`

This file is where infrastructure clients are initialized from environment settings.

### `app/db/init_db.py`

Contains startup seeding logic.

Right now it has one job:

- create a default admin user if one does not already exist

It uses:

- `get_settings()` for seed credentials
- `hash_password()` to store the password securely
- `UserRepository` to check and insert the user

This file is called from `app/main.py` during startup.

## 7. ORM models: `app/models/`

These files define the database tables and ORM relationships.

### `app/models/__init__.py`

Re-exports the main model classes:

- `InventoryItem`
- `StockMovement`
- `User`

This helps Alembic and imports elsewhere.

### `app/models/user.py`

Defines:

- `UserRole` enum
- `User` ORM model

Fields include:

- `email`
- `full_name`
- `hashed_password`
- `role`
- `is_active`
- timestamps

Relationship:

- one user can have many `stock_movements`

This table powers authentication and role-based access control.

### `app/models/inventory_item.py`

Defines the `InventoryItem` model.

Fields include:

- `sku`
- `name`
- `description`
- `quantity`
- `reorder_level`
- `unit_price`
- timestamps

Relationship:

- one item can have many stock movement records

This is the core inventory table.

### `app/models/stock_movement.py`

Defines:

- `MovementType` enum with `in_` and `out`
- `StockMovement` model

Fields include:

- `item_id`
- `quantity`
- `movement_type`
- `note`
- `performed_by`
- `created_at`

Relationships:

- linked to `InventoryItem`
- linked to `User`

This table provides audit/history of inventory changes.

## 8. Schemas: `app/schemas/`

These are Pydantic models used for request validation and response serialization.

### `app/schemas/__init__.py`

Marks the schema folder as a package.

### `app/schemas/common.py`

Defines `ORMModel`, a reusable base schema with:

- `from_attributes=True`

That lets Pydantic read SQLAlchemy model objects directly.

### `app/schemas/auth.py`

Defines authentication-related schemas.

Includes:

- `LoginRequest`
- `TokenResponse`

Note:

- `LoginRequest` exists, but the actual login route currently uses `OAuth2PasswordRequestForm` instead

### `app/schemas/user.py`

Defines:

- `UserCreate`
- `UserRead`

Purpose:

- validate user registration input
- shape user data returned from the API

Note:

- there is currently no dedicated `UserDelete` or `UserDeactivate` request schema
- the delete route takes `full_name` directly as a query parameter

### `app/schemas/item.py`

Defines:

- `ItemCreate`
- `ItemUpdate`
- `ItemRead`

This file contains validation rules such as:

- SKU length
- item name length
- non-negative quantity
- positive unit price

### `app/schemas/movement.py`

Defines:

- `StockMovementCreate`
- `StockMovementRead`

This validates stock movement input and shapes movement history responses.

## 9. Repository layer: `app/repositories/`

Repositories isolate direct database queries from the service layer.

### `app/repositories/__init__.py`

Marks the folder as the repository package.

### `app/repositories/user_repository.py`

Encapsulates user table queries.

Methods:

- `get_by_email()`
- `get_by_id()`
- `list_users()`
- `create()`
- `delete()`

This is the main persistence access point for user-related service logic.
  
### `app/repositories/item_repository.py`

Encapsulates inventory item queries.

Methods:

- `get_by_id()`
- `get_by_sku()`
- `list_items()`
- `create()`
- `update()`

Important detail:
  
- `update()` assumes the SQLAlchemy model has already been modified in memory, then commits and refreshes it

### `app/repositories/movement_repository.py`

Encapsulates stock movement database access.

Methods:

- `create()`
- `list_for_item()`

This repository is mainly used by inventory movement business logic.

## 10. Service layer: `app/services/`

Services contain business rules. They sit between the API routes and repositories.

### `app/services/__init__.py`

Marks the folder as the service package.

### `app/services/auth_service.py`

Handles authentication and user registration logic.

Responsibilities:

- reject duplicate emails
- hash passwords before saving
- verify credentials during login
- reject inactive users
- create JWT access tokens

This file owns auth business rules, not the route file.

### `app/services/user_service.py`

Handles user-related business logic.

Right now it is intentionally small:

- `get_user_or_404()`
- `list_users()`
- `get_user()`
- `delete_user()`

It acts as a clean abstraction layer even though the logic is simple for now.

Current delete behavior:

- looks up the user by id
- checks that `full_name` matches
- checks that the user is active
- sets `is_active = False`
- then calls the repository delete method

Important review note:

- the service currently performs a soft-delete style state change (`is_active = False`)
- but the repository method is named `delete()` and performs a real database delete
- this mismatch should be reviewed carefully because `stock_movements.performed_by` references `users.id`

### `app/services/item_service.py`

Handles inventory item business logic and Redis caching.

Main responsibilities:

- list items with Redis caching
- get a single item with Redis caching
- create items
- update items
- invalidate cache when data changes
- enforce SKU uniqueness
- raise 404 or 409 app errors where appropriate

Cache keys used here:

- `inventory:items:all`
- `inventory:item:{item_id}`

This file is where item read performance is improved through caching.

### `app/services/inventory_service.py`

Handles stock movement business rules.

Main responsibilities:

- ensure the item exists
- prevent stock from going below zero on `out` movements
- increase or decrease item quantity
- create a movement record
- invalidate related Redis cache
- trigger a Celery low-stock alert task when quantity reaches reorder level

This is one of the most business-critical files in the project.

## 11. Tests: `tests/`

This folder contains the automated unit tests for the service layer.

The current suite is designed to be fast and isolated:

- repositories are mocked
- Redis is mocked
- no real PostgreSQL or Redis connection is needed
- the tests focus on business logic in `app/services/`

### `tests/conftest.py`

Provides shared test fixtures and bootstrap logic.

Responsibilities:

- adds the project root to `sys.path` so `app.*` imports work in pytest
- forces `DEBUG=false` for test collection to avoid local `.env` parsing issues
- defines factories for:
  - `User`
  - `InventoryItem`
  - `StockMovement`
- defines a fake async Redis mock used by service tests

### `tests/test_auth_service.py`

Unit tests for `AuthService`.

Current coverage includes:

- successful user registration
- duplicate-email rejection
- invalid login rejection
- inactive-user login rejection

### `tests/test_user_service.py`

Unit tests for `UserService`.

Current coverage includes:

- fetching a user successfully
- handling missing users
- delete flow behavior
- full-name mismatch rejection
- inactive-user rejection during delete

### `tests/test_item_service.py`

Unit tests for `ItemService`.

Current coverage includes:

- reading item lists from cache
- handling missing items
- rejecting duplicate SKUs
- updating item fields and invalidating cache

### `tests/test_inventory_service.py`

Unit tests for `InventoryService`.

Current coverage includes:

- missing-item rejection during stock movement
- insufficient-stock rejection
- successful quantity updates
- cache invalidation
- low-stock alert trigger behavior

### Test execution

The current test command is:

```powershell
poetry run pytest -q
```

At the moment, the implemented tests are unit tests only. They do not yet cover:

- API endpoint integration
- real database integration
- real Redis integration
- end-to-end auth flows through HTTP requests
## 12. Background tasks: `app/tasks/`

This folder contains Celery setup and background jobs.

### `app/tasks/__init__.py`

Marks the tasks folder as a package.

### `app/tasks/celery_app.py`

Creates and configures the shared Celery app.

It sets:

- app name
- Redis broker
- Redis backend
- included task modules
- JSON serialization settings
- UTC timezone behavior

All Celery workers rely on this object.

### `app/tasks/inventory_tasks.py`

Defines the background task:

- `send_low_stock_alert`

Right now it only writes a warning log when stock drops to or below reorder level.

So the project already has the background task plumbing, but the alert action itself is currently just logging, not email/SMS/etc.

## 13. Alembic migration files: `alembic/`

This folder manages database schema history.

### `alembic/env.py`

This is the runtime configuration file Alembic uses when migrations run.

It:

- loads app settings
- injects the sync database URL into Alembic
- imports model modules so metadata is registered
- exposes `Base.metadata` as migration target metadata
- supports both offline and online migration modes

This file is what connects your app models to Alembic migration generation/execution.

### `alembic/script.py.mako`

This is the template Alembic uses when generating new migration files.

When you run a command like:

```powershell
alembic revision -m "add something"
```

Alembic uses this template to create the new migration skeleton.

### `alembic/versions/20260323_000001_initial_schema.py`

This is the first migration in the project.

It creates:

- `users`
- `inventory_items`
- `stock_movements`

It also creates:

- indexes
- enum types for roles and movement type
- foreign keys between tables

Its `downgrade()` function reverses those changes.

## 14. Generated and cache files you can usually ignore

### `__pycache__/` folders

These exist in many directories such as:

- top-level `__pycache__/`
- `app/.../__pycache__/`
- `alembic/.../__pycache__/`

They contain compiled `.pyc` bytecode generated by Python.

These are not handwritten source files and are not part of the business logic.

### `*.pyc` files

These are compiled Python cache files generated automatically.

They should not be edited manually.

## 15. How the pieces work together

Here is the practical architecture of this project:

### Startup

- `app/main.py` starts the app
- `app/core/logging.py` configures logs
- `app/core/config.py` loads environment settings
- `app/db/session.py` creates DB and Redis clients
- `app/db/init_db.py` seeds the default admin user

### Auth flow

- `app/api/v1/auth.py` receives login/register requests
- `app/services/auth_service.py` validates business rules
- `app/core/security.py` hashes passwords and creates JWTs
- `app/repositories/user_repository.py` persists user data

### User flow

- `app/api/v1/users.py` handles `/me`, list, get-by-id, and delete requests
- `app/services/user_service.py` contains user lookup and delete rules
- `app/repositories/user_repository.py` reads and deletes user rows
- `app/models/user.py` defines the underlying user table and role enum

### Item flow

- `app/api/v1/items.py` handles item endpoints
- `app/services/item_service.py` manages validation and caching
- `app/repositories/item_repository.py` reads/writes items
- `app/models/inventory_item.py` defines the underlying table
- `app/schemas/item.py` validates request/response data

### Inventory movement flow

- `app/api/v1/inventory.py` accepts stock movement requests
- `app/services/inventory_service.py` updates quantity and creates history
- `app/repositories/item_repository.py` updates the item
- `app/repositories/movement_repository.py` creates the movement row
- `app/tasks/inventory_tasks.py` logs a low-stock alert asynchronously if needed

### Test flow

- `tests/conftest.py` provides shared factories and mocks
- `tests/test_*` files call service methods directly
- `pytest` runs async unit tests with mocked collaborators

## 16. Which files matter most when making changes

If you want to change a specific feature, these are the main files to look at:

- authentication: `app/api/v1/auth.py`, `app/services/auth_service.py`, `app/core/security.py`, `app/models/user.py`
- user listing and roles: `app/api/v1/users.py`, `app/core/dependencies.py`, `app/models/user.py`
- item CRUD: `app/api/v1/items.py`, `app/services/item_service.py`, `app/repositories/item_repository.py`, `app/schemas/item.py`
- stock movement: `app/api/v1/inventory.py`, `app/services/inventory_service.py`, `app/models/stock_movement.py`, `app/schemas/movement.py`
- startup/config: `app/main.py`, `app/core/config.py`, `app/db/session.py`, `.env`
- database schema: `app/models/`, `alembic/env.py`, `alembic/versions/...`
- background jobs: `app/tasks/celery_app.py`, `app/tasks/inventory_tasks.py`, `celery_worker.py`
- unit tests: `tests/conftest.py`, `tests/test_auth_service.py`, `tests/test_user_service.py`, `tests/test_item_service.py`, `tests/test_inventory_service.py`

## 17. Summary

This project is organized in a clean layered backend structure:

- `api` handles HTTP
- `core` handles shared infrastructure concerns
- `db` handles setup and sessions
- `models` define tables
- `schemas` define request/response shapes
- `repositories` talk directly to the database
- `services` hold business logic
- `tasks` handle async background work
- `alembic` manages schema evolution
